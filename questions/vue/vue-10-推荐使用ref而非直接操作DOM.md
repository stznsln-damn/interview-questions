# 在 Vue 中为什么推荐使用 ref 而非直接操作 DOM？

## 核心要点

- Vue 是**数据驱动视图**的框架，视图应由数据和声明式模板决定；直接操作 DOM 会绕过 Vue 的更新机制，导致状态被覆盖、难以维护，且在 SSR 下不可用。
- **模板 ref** 是“需要拿到 DOM 或组件实例”时的官方方式：与挂载/卸载生命周期一致，不破坏响应式更新，SSR 下可在 `onMounted` 中使用，便于维护。
- 能通过数据与指令表达的尽量用数据；必须操作 DOM 时（焦点、滚动、测量、第三方库）用 ref 拿节点再操作。

## 详细解释

### 直接操作 DOM 的问题

- **与 Vue 更新冲突**：Vue 根据数据通过虚拟 DOM 更新真实 DOM。用 `document.querySelector`、`innerHTML` 等直接改 DOM，Vue 不知情，下一轮更新可能覆盖你的修改，出现闪烁或错误。
- **生命周期与时机**：在 `setup`/`created` 里查 DOM，节点可能尚未挂载（如 `v-if`）；组件销毁后若仍操作旧节点，会操作已脱离文档的节点。直接 DOM 与“何时挂载、何时销毁”无绑定。
- **SSR**：服务端无 `document`/`window`，直接 DOM 代码会报错或需大量环境判断。
- **可维护性**：同一界面既有 Vue 管理又有手改 DOM，逻辑分散，难以理清谁在改、以谁为准。

### 使用 ref 的好处

- **与组件/节点同生命周期**：模板 `ref="xxx"` 配合 script 中 `ref(null)`，挂载后 Vue 赋值为真实 DOM 或组件实例，卸载或 `v-if` 移除时置为 `null`，拿到的始终是当前有效节点。
- **不破坏响应式更新**：ref 只是“访问 DOM 的句柄”，展示仍由模板 + 数据驱动；在 `onMounted` 或事件里用 `xxxRef.value` 做 focus、滚动、测量等，不与 Vue 的 diff 冲突。
- **SSR 友好**：在 `onMounted` 中使用 ref，服务端不执行 DOM 逻辑，客户端水合后 ref 才有值。
- **风格统一**：需要 DOM 时统一用 ref + 生命周期/事件，便于协作和审查。

### 何时可以“动”DOM

能通过 `v-if`、`:class`、`:style`、表单绑定等表达的，优先用数据。只有在必须接触真实 DOM 时再用 ref，例如：焦点控制、滚动位置、尺寸测量、集成非 Vue 库（图表、地图等）。此时用 ref 拿节点再操作，既满足需求又不破坏“Vue 负责渲染、ref 负责访问”的分工。

## 总结

1. 数据驱动视图，直接改 DOM 易与 Vue 更新冲突且难维护、不利于 SSR。
2. ref 与挂载/卸载同步，时机明确，不破坏响应式，SSR 下可在 onMounted 中使用。
3. 能数据驱动的用数据；必须操作 DOM 时用模板 ref 拿节点再操作，而不是 document.querySelector 等直接操作。
