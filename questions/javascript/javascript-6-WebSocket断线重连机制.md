# 如何实现 WebSocket 的断线重连机制？

## 核心要点

- 在连接**关闭（onclose）或错误（onerror）**时判断是否需要重连；**正常关闭**（关闭码 1000 或业务主动 close）不重连，避免与业务冲突。
- 使用**延迟 + 退避**（如指数退避）和**最大重试次数**，避免立刻重连和无限重连；每次重连**新建 WebSocket 实例**并重新绑定事件。
- 可选：**心跳**检测“假死”连接，超时后主动关闭或触发重连，由同一套重连逻辑处理。

## 详细解释

### 为什么需要断线重连

WebSocket 可能因网络抖动、服务重启、代理/中间设备超时、长时间无数据被断开等而关闭。不做重连时用户只能刷新页面；重连可在断线后自动恢复连接，提升体验。

### 实现要点

1. **监听断开与错误**：在 `onclose` 中统一判断是否重连；若环境存在只 `error` 不 `close` 的情况，可在 `onerror` 里也触发同一套重连入口，并注意避免与 `onclose` 重复执行。
2. **区分正常关闭**：通过 `event.code === 1000` 或业务约定码、以及“主动关闭”标记（如 `manualClose`）判断，只有异常断开才重连。
3. **重试延迟与退避**：设置初始延迟和最大延迟；采用指数退避（如 delay = min(初始 × 2^n, 最大)）或线性递增，避免在服务不可用时频繁建连。
4. **重试次数上限**：超过最大次数后停止重连，可提示用户“连接失败，请刷新重试”。
5. **重建连接**：每次重连 `new WebSocket(url)` 并重新绑定 onopen/onmessage/onclose/onerror；重连成功后重置重试次数与延迟。使用 setTimeout 做单次延迟，注意清理旧实例和定时器，避免多个重连定时器叠加。

### 心跳（可选）

长时间无数据时，部分网络不会立刻触发 close。可定时发心跳（ping 或业务心跳包），若超时未收到应答则认为“假死”，主动 `ws.close()` 或直接触发重连，由统一的重连逻辑处理。

## 总结

1. onclose（及按需 onerror）中判断是否重连；正常关闭不重连。
2. 延迟 + 退避 + 最大重试次数；每次重连新建实例并重新绑定事件。
3. 可选心跳检测假死，超时后触发重连。
