# Vue 的 Diff 为何采用同层比较，而非跨层？

## 核心结论

这是性能与收益的权衡决策。跨层比较理论上能找到更精确的最小 DOM 操作，但算法复杂度是 O(n³)，远超同层比较的 O(n)。实际前端开发中，跨层移动节点的场景极其罕见，同层比较能以极低成本覆盖绝大多数情况。

## 跨层比较为什么不可行

### 1. 算法复杂度

传统树 Diff 的经典算法复杂度是 O(n³)：

- 遍历旧树所有节点 × 新树所有节点 = O(n²)
- 每对节点还要计算子树编辑操作 = 再乘 O(n)

1000 个 VNode 需约 10 亿次比较，在 16ms 帧预算内不可能完成。

### 2. DOM 操作本身的代价

即使找出跨层移动，浏览器没有原生的高效跨层移动操作，本质还是 removeChild + appendChild，与销毁重建成本差距不大。

## 同层比较为什么足够

### 1. 前端 UI 的变更模式

实际业务中 DOM 变更绝大多数是：同层节点增/删/排序、属性/文本更新、条件渲染导致的同层替换。跨层移动几乎不存在。

### 2. 复杂度降为 O(n)

同层比较逐层遍历，每个节点最多访问一次，千级节点在 1ms 内完成。

### 3. 错判代价可接受

跨层移动被降级为"旧位置销毁 + 新位置创建"，发生频率极低，多出的开销远小于每次跨层比较的常态开销。

## 复杂度对比

| 策略      | 时间复杂度 | 1000 节点耗时量级 | 覆盖场景 |
| --------- | ---------- | ----------------- | -------- |
| 跨层 Diff | O(n³)      | ~10⁹ 次运算       | 100%     |
| 同层 Diff | O(n)       | ~10³ 次运算       | >99%     |

## Vue Diff 同层比较的具体策略

```text
旧 VNode 树          新 VNode 树

    A                    A          ← 同层比较 A vs A
   / \                 / | \
  B   C              B   D   C     ← 同层比较 [B,C] vs [B,D,C]
 / \                 /
E   F               E              ← 同层比较 [E,F] vs [E]
```

每一层子节点列表内部，Vue3 使用最长递增子序列（LIS）算法优化移动操作，尽可能复用和减少 DOM 操作。

## 总结

1. **跨层 Diff 复杂度 O(n³) 不可接受**，无法满足前端高频更新的性能要求
2. **前端 UI 变更模式决定跨层移动极其罕见**，同层比较覆盖 99% 以上真实场景
3. **错判代价可控**：跨层移动降级为"删除+新建"，频率低、影响小
4. **同层 O(n) 复杂度**配合 Vue3 LIS 优化，在绝大多数场景下接近最优
