# 有一个需要支持 10 万条数据渲染的表格组件，你会如何设计和优化？

## 核心思路

绝对不能把 10 万个 DOM 节点全量渲染到页面。核心方案是虚拟滚动（Virtual Scroll），辅以多种优化手段。

## 一、核心方案：虚拟滚动

只渲染可视区域内的行（20~30 行），通过监听滚动事件动态替换渲染内容。

```text
┌──────────────────────────┐
│      padding-top         │  ← 空白撑起上方不可见区域
├──────────────────────────┤
│  Row 501 ~ Row 530       │  ← 实际只渲染 20~30 个 DOM 行
├──────────────────────────┤
│      padding-bottom      │  ← 空白撑起下方不可见区域
└──────────────────────────┘
```

核心计算：

- 容器总高度 = 行数 × 行高
- 起始索引 = Math.floor(scrollTop / rowHeight)
- 可见条数 = Math.ceil(containerHeight / rowHeight)
- 偏移量 = startIndex \* rowHeight（用 transform: translateY() 定位）

| 类型     | 适用场景   | 复杂度                       |
| -------- | ---------- | ---------------------------- |
| 固定行高 | 行高统一   | 简单，乘除法计算             |
| 动态行高 | 行高不一致 | 需维护位置缓存表，预估后修正 |

## 二、辅助优化手段

### 1. 缓冲区（Overscan）

可视区域上下各多渲染几行，避免快速滚动白屏闪烁。

### 2. 滚动性能

- 用 requestAnimationFrame 做节流
- 使用 transform 而非 top/margin-top 定位，避免触发 layout
- 设置 will-change: transform 提前创建合成层

### 3. 列维度虚拟化

列数多时，水平方向也需虚拟化。

### 4. 单元格渲染优化

- 使用轻量函数式组件
- 复用 DOM 节点（对象池模式）
- 复杂列懒渲染

### 5. 数据层优化

- Web Worker 处理排序、过滤等计算
- 树形结构提前扁平化缓存

### 6. 固定列/表头

使用 position: sticky 或独立容器同步滚动。

## 三、整体架构设计

```text
TableComponent
├── 数据层：shallowRef 存储原始数据，Worker 处理计算
├── 虚拟滚动层：滚动监听、可视范围计算、占位容器
├── 渲染层：固定表头、固定列、单元格渲染
└── 交互层：选中状态（Set 存储 id）、编辑态、展开收起
```

## 四、数据响应式注意事项

- 用 shallowRef 存储原始数据，避免深层代理
- 只对 visibleData（可见的 20~30 条）做响应式
- 选中状态用独立 Set/Map 管理，不挂在行数据上

## 五、成熟方案参考

| 库             | 特点                       |
| -------------- | -------------------------- |
| vxe-table      | Vue 生态，原生支持虚拟滚动 |
| AG Grid        | 企业级方案，框架无关       |
| TanStack Table | Headless 设计，UI 自定义   |

## 总结

1. **核心方案是虚拟滚动**：只渲染可视区域 20~30 行 DOM，滚动时动态替换
2. **滚动性能**：transform 定位、requestAnimationFrame 节流、缓冲区防白屏
3. **数据层**：shallowRef 避免深层代理，计算密集操作放 Web Worker
4. **架构分层**：数据层、虚拟滚动层、渲染层、交互层各司其职
5. **实际项目**：优先考虑成熟方案，避免重复造轮子
