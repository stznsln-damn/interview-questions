## 概述

执行完整的代码提交工作流：格式校验 -> 暂存 -> 更新统计 -> 生成日志 -> 提交 -> 推送。

## 工作流程

### 第一步：预检查

并行执行 `git status` 和 `git diff --stat`，了解当前变更情况。

- 如果没有任何变更（无未跟踪文件且无修改），通知用户并终止流程。
- 如果存在可能包含敏感信息的文件（.env、credentials.json 等），向用户发出警告。

### 第二步：格式校验

提交前执行 lint 和格式化检查：

```bash
pnpm lint:fix && pnpm format
```

- 如果存在无法自动修复的错误，向用户展示错误信息并询问如何处理。
- 如果自动修复了文件，继续下一步（修复后的文件将包含在提交中）。

### 第三步：暂存变更

执行 `git add .` 暂存所有变更。

- 如果用户指定了特定文件，则只暂存指定的文件。

### 第四步：更新 README 题目统计

扫描 `questions/` 目录下的所有 `.md` 文件，按文件名前缀（`-` 前的部分）统计各分类的题目数量，然后更新 `README.md` 中 `## 题目分类` 下的表格。

**统计规则**：

- 文件名格式为 `{分类}-{序号}-{描述}.md`，提取 `{分类}` 作为分类标识
- 分类名首字母大写（如 `vue` → `Vue`，`javascript` → `JavaScript`）
- 按数量降序排列；数量相同时按分类名字母序

**表格格式**（保持与现有 README 一致）：

```markdown
| 分类 | 数量 | 说明                              |
| ---- | ---- | --------------------------------- |
| Vue  | 4    | Vue 生命周期、响应性、Diff 算法等 |
```

**说明列生成规则**：

- 读取该分类下所有文件的标题（文件名中 `{描述}` 部分），取前 3 个用中文顿号 `、` 连接，超过 3 个在末尾加 `等`

**注意**：仅在表格内容实际发生变化时才更新文件。如果统计结果与现有表格一致，跳过此步。

### 第五步：生成更新日志

在 `changelogs/` 目录下为本次提交生成一个更新日志文件。

**文件命名**：`{YYYY-MM-DD}_{HH-mm-ss}.md`（使用当前时间，精确到秒，避免冲突）

**文件内容格式**：

```markdown
# 更新日志 {YYYY-MM-DD HH:mm:ss}

## 变更文件

- `path/to/file1.md`（新增）
- `path/to/file2.md`（修改）
- `path/to/file3.md`（删除）

## 提交信息

{即将使用的 commit message}

## 题目统计

| 分类 | 数量 |
| ---- | ---- |
| Vue  | 4    |
| ...  | ...  |

共计 {总数} 道面试题
```

**规则**：

- `changelogs/` 目录不存在时自动创建
- 变更文件列表从 `git diff --cached --name-status` 获取，标注状态（新增/修改/删除/重命名）
- 题目统计部分复用第四步的统计结果
- 日志文件生成后需一并加入暂存区

### 第六步：重新暂存

将第四步和第五步中新增或修改的文件（README.md、changelogs/\*.md）加入暂存区：

```bash
git add README.md changelogs/
```

### 第七步：生成提交信息

分析所有暂存的变更，按照 Conventional Commits 规范生成提交信息。

**类型规则**（来自本项目的 commitlint 配置）：

- `feat` — 新增面试题或功能
- `fix` — 修复内容错误
- `docs` — 文档变更（README、AGENTS.md 等）
- `style` — 格式调整（不影响内容）
- `refactor` — 内容重构
- `chore` — 工具或配置变更

**格式**：`<type>: <简洁的中文描述>`

**示例**：

- `feat: 新增 Vue Diff 算法面试题`
- `docs: 更新 README 文档`
- `chore: 添加 markdownlint 配置`

### 第八步：确认提交

向用户展示以下信息：

1. 待提交的文件列表
2. 生成的提交信息

等待用户确认或修改提交信息后再继续。

### 第九步：提交并推送

```bash
git commit -m "<提交信息>"
git push
```

- 如果提交因 pre-commit hook（lint-staged）失败，修复问题后用新的提交重试。
- 如果推送失败，展示错误信息并建议解决方案（如落后于远程分支时先执行 pull）。
- 推送成功后，执行 `git status` 展示最终状态。

## 重要规则

- 禁止使用 `git commit --amend`，除非用户明确要求。
- 禁止使用 `git push --force`。
- 禁止使用 `--no-verify` 跳过钩子。
- 禁止修改 git 配置。
- 提交信息的描述部分必须使用中文。
- 提交前必须经过用户确认。
